# Фаза 2 — Дизайн: укрепление UI с сохранением эстетики (дизайн-документ)

> Дата: 2026-06-16 · Статус: на согласовании · Автор: Claude Code
> Предшествует: Фаза 1 «Надёжность + Качество» (влита в `main`, merge `8ed3557`).

## 1. Цель и контекст

Дашборд функционально зрелый и стилистически цельный (тёмная «кибер»-эстетика). Слабые места —
не во внешнем виде, а в инженерии UI: вёрстка фактически десктоп-only (1 медиа-запрос), нет
доступности, таблица из 613 задач рендерится разом, весь UI — монолит `index.html` (2019 строк).

**Цель Фазы 2 — укрепить «как сделано», не трогая эстетику:** адаптив на телефон/десктоп/крупные
экраны, доступность, производительность таблицы, модульность кода, дизайн-система в Claude Design.

## 2. Зафиксированные решения (брейнсторм + визуальные макеты)

- **Эстетика — без изменений:** палитра (`#06060f` / `#00e5ff` / `#b44fff` / `#00ff9d` / …),
  шрифты Orbitron + Exo 2, тёмный фон. Меняем структуру и поведение, не внешний вид.
- **Устройства:** десктоп (основное) + крупные экраны (совещания/проектор) + телефон (владелец
  часто смотрит с телефона) → **полноценный адаптив на три класса экранов**.
- **Мобильная таблица «Задачи» → карточки** (вариант A). На десктопе/крупных — таблица как сейчас.
- **Мобильная навигация → нижняя панель** (bottom bar, вариант B). На десктопе — вкладки сверху как сейчас.
- **Модуляризация — ES-модули по функциям** (вариант «модули по функциям»).

## 3. Объём и под-фазы

Реализуем последовательно; каждая под-фаза — самостоятельный рабочий результат и отдельный набор коммитов.

| Код | Под-фаза | Суть |
|---|---|---|
| **D4** | Модуляризация | вынести CSS + разбить JS на ES-модули (без изменения логики) |
| **D1** | Адаптив + доступность | 3 брейкпоинта + a11y на чистой модульной базе |
| **D2** | Виртуализация | оконный рендер таблицы/карточек задач |
| **D3** | Дизайн-система | каталог компонентов в Claude Design (claude.ai/design) |

Порядок **D4 → D1 → D2 → D3**: сперва чистая структура, на ней безопаснее адаптив и виртуализация;
дизайн-система фиксирует уже адаптивные и доступные компоненты.

### D4 — Модуляризация (ES-модули по функциям)

Целевая структура:
```
index.html            — разметка-оболочка + <link css/styles.css> + <script type="module" src="js/main.js">
css/styles.css        — извлечённые стили (позже — медиа-запросы D1)
js/config.js          — CONFIG + дефолты (year/hoursPerUnit/redmineBase), чтение data.config
js/helpers.js         — escapeHTML, parseDate, byDeadline, getQuarter, getProgressColor/getGaugeColor,
                        PROJECT_PALETTE, CLOSED_STATUSES, PRIORITY_STAR, Q_COLORS
js/state.js           — общее состояние (allProjects/allTasks/allTasks2026/allDetails/allCurators)
js/render/overview.js — KPI, статус-бейджи, таблица «Проекты — Готовность», вкладка Обзор
js/render/vysv.js     — блок «Высвобождение %» (renderVysvBlock, _vysvDetailHTML, гейджи)
js/render/deadlines.js— тепловая карта дедлайнов
js/render/tasks.js    — renderTasks, фильтры задач
js/render/detail.js   — loadDetail (детализация проекта)
js/render/projects.js — renderFullProjTable
js/filters.js         — applyExecFilters, filterTasksByStat, populate*, highlightGauge
js/main.js            — loadData → initDashboard → setupEventListeners (точка входа)
```
**Принцип:** чистое извлечение, **логика не меняется**. Приёмка под-фазы — дашборд визуально и по
цифрам идентичен (36/613/5, 17 просрочено, 122%).

> **ES-модули и `file://`:** модули требуют http(s) (CORS) — открывать через сервер (у нас уже есть
> локальный; на Amvera раздаётся по http). Открытие `index.html` как файла перестанет работать — это
> приемлемо (дашборд и так смотрят по http). Фиксируем явно.

### D1 — Адаптив + доступность

**Брейкпоинты:**
- **Телефон (≤ 640px):** одна колонка; KPI стопкой; гейджи кураторов стопкой; таблицы «Задачи» и
  «Проекты» → карточки; навигация → **нижняя панель**; детализация (3 колонки) → стопка; фильтры — на всю ширину.
- **Десктоп (641–1599px):** как сейчас (поведение не меняется).
- **Крупные (≥ 1600px):** ограничение `max-width` контейнера + чуть крупнее типографика/отступы для
  читаемости с расстояния (совещания/проектор).

**Доступность (a11y):**
- Роли вкладок: `role="tablist"/"tab"/"tabpanel"`, `aria-selected`, управление с клавиатуры (стрелки/Tab).
- `label`/`aria-label` на `select`-фильтрах и кнопках; осмысленные имена интерактивных элементов.
- Видимые фокус-стили `:focus-visible` в теме (cyan-обводка), логичный порядок табуляции.
- Контраст по WCAG AA: проверить `--text-dim` (`rgba(.5)`) и бейджи; поднять где не проходит.
- `aria-live="polite"` на динамически обновляемых блоках (KPI-карточки, счётчики задач).
- SVG-гейджи: `role="img"` + `aria-label` с процентом высвобождения.
- Семантика: `nav` / `main` / `section`, корректные уровни заголовков.

**Проверка:** скилл `web-design-guidelines` (аудит UI-кода) + ручная проверка на трёх брейкпоинтах.

### D2 — Виртуализация таблицы «Задачи»

**Проблема:** 613 строк (270 активных) генерируются и вставляются в DOM разом — лаги при больших объёмах.

**Подход:** оконный рендер (virtual scroll) — в DOM только видимые строки + буфер; общая высота
держится спейсером; пересчёт окна при скролле (scroll handler / `IntersectionObserver`). На телефоне
(карточки) — аналогичный оконный рендер по карточкам. Vanilla JS, без библиотек.
**Альтернатива** (проще, на случай сложностей): пагинация по N строк. **Выбор по умолчанию — оконный
рендер** (сохраняет привычный единый скролл).

**Приёмка:** при 600+ задачах прокрутка плавная; фильтры (статус/проект) и ссылки работают; счётчики верны.

### D3 — Дизайн-система в Claude Design

Каталогизировать компоненты (в текущей эстетике, уже адаптивные/доступные после D1) как
переиспользуемые, с превью и спецификацией состояний; синхронизировать на **claude.ai/design** через
`DesignSync` (скилл `/design-sync`), по одному компоненту, не «оптом».

**Компоненты:** KPI-карточка / stat-pill · гейдж (полукруг SVG) · прогресс-бар · статус-бейдж ·
дедлайн-чип · project-tag · строка таблицы / карточка задачи (mobile) · квартальный бейдж · вкладки
(desktop) + нижняя панель (mobile) · filter-select · info-block детализации · приоритет-звезда.

**Замечания:** D3 — после D1 (компоненты финальные). DesignSync требует claude.ai-логина; если в сессии
его нет — `/design-login`. Дизайн-система — справочник/каталог, **не** рантайм-источник дашборда
(реализация остаётся в модульных CSS/JS).

## 5. Тестирование

- Пайплайн-тесты Фазы 1 (`tests/`, 22 шт.) остаются зелёными.

- Ручная визуальная проверка на трёх брейкпоинтах (телефон / десктоп / крупный) + аудит

- Контрольная сверка «тот же вид/цифры» после D4 и после D1.
- (jsdom-юнит-тесты UI не делаем — нет доступа к сети для установки в текущей среде.)

## 6. Риски и обратная совместимость

- **D4 — крупный механический рефактор** (~1000 строк JS): риск регрессий. Митигируем поэтапным
  извлечением (модуль за модулем) + сверкой идентичности после каждого шага.
- **ES-модули требуют http** — локально сервер есть, на Amvera http; открытие как `file://` отпадёт (ок).
- **D3 требует claude.ai-доступа** (DesignSync); при отсутствии — `/design-login` или отложить D3.
- Эстетика и данные не меняются — пользовательские цифры и вид остаются прежними на десктопе.
