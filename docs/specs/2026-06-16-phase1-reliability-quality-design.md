# Фаза 1 — Надёжность + Качество (дизайн-документ)

> Дата: 2026-06-16 · Статус: на согласовании · Автор: Claude Code (по итогам диагностики проекта)

## 1. Цель и контекст

Дашборд функционально зрелый, но **хрупкий по инженерии**: пайплайн `extract_data.py → data.json → index.html`
ломается молча при изменении исходных Excel-файлов, изменения в коде ничем не подстрахованы (0 тестов / 0 CI / 0 линтеров).

Цель Фазы 1 — **застраховать основу**: ни один сбой не проходит молча, изменения становятся безопасными.
Это фундамент перед Фазой 2 (дизайн) и треком «Автоматизация».

**Принцип:** ничего не ломаем в текущем поведении дашборда. Все изменения — аддитивные и обратносовместимые.
Старый формат данных и существующая логика рендера остаются рабочими.

## 3. Детали по пунктам

### R3 — Единый конфиг (делаем первым: на него опираются R1, R2, R5)

Новый файл **`config.py`** — единственный источник правды для «магических» значений:

```python
YEAR = 2026                       # отчётный год дашборда
REDMINE_BASE = 'https://transformation.rm.mosreg.ru/#/issues'
HOURS_PER_UNIT = 1972             # часов в 1 шт.ед.
TOTAL_CURATOR = 'Комитет и РЦТ'
CURATOR_ORDER = ['Кренёва АА', 'Родевальд СЕ', 'Гуляев ВА', 'Кудряшов ЕС']
CLOSED_STATUSES = {'Закрыта', 'Закрыто', 'Выполнено', 'Выполнена', 'Завершена'}
CURATOR_NAME_FIX = {'Кудряшев': 'Кудряшов'}
COLUMN_SYNONYMS = { ... }         # см. R2
```

- Python-скрипты (`extract_data.py`, `process_report.py`, `overdue_report.py`) импортируют значения из `config.py`
  вместо локальных дублей (`HOURS_PER_UNIT` в `process_report.py:425`, `REDMINE_BASE` в `overdue_report.py:26` и т.д.).
- **Прокидывание в HTML без второго fetch:** `extract_data.py` пишет подмножество конфига в `data.json`
  под ключом **`config`**: `{ "year": 2026, "redmine_base": "...", "hours_per_unit": 1972 }`.
  `index.html` читает `data.config` и заменяет хардкод: `'2026'` (строки 817, 1686, 1710, 1771, 1838, 1881)
  и `1972` (1230, 1482, 1484, 1490). Если ключа `config` нет (старый `data.json`) — фолбэк на текущие
  захардкоженные значения, чтобы не сломать обратную совместимость.

### R2 — Гибкий матчинг колонок

Обобщаем существующий `cell_any` в резолвер колонок в `config.py`/`extract_data.py`:

- `normalize_col(name)` — strip, lowercase, схлопывание пробелов, `ё→е`.
- `resolve_column(df, *candidates) -> str | None` — ищет первую колонку из синонимов по нормализованному имени.
- `require_column(df, key, *candidates) -> str` — то же, но при отсутствии бросает понятную ошибку:
  `"Колонка '<key>' не найдена. Ожидалась одна из: [...]. В файле есть: [...]"`.
- Применяется к обязательным колонкам (`Трекер`, `#`) и к колонкам высвобождения/штатки
  (уже частично покрыто `cell_any` — переводим на общий механизм + словарь `COLUMN_SYNONYMS`).

### R1 — Валидация выгрузки + атомарная запись

В конце `extract()`, **перед записью** `data.json`:

1. **Структурные проверки:** обязательные колонки присутствуют (через `require_column`); трекеры
   «Паспорт проекта» / «Мероприятие проекта» дали непустые наборы.
2. **Sanity-проверки счётчиков:** `projects > 0`, `all_tasks > 0`, `curators` непуст.
3. **Регрессия против предыдущего `data.json`** (если есть): число проектов/задач не упало до 0 и не
   просело больше, чем на порог (по умолчанию 50%) — иначе предупреждение/стоп (порог в `config.py`).
4. **Атомарная запись:** сериализация в `data.json.tmp` → валидация JSON-парсингом → `os.replace(tmp, data.json)`.
   При любой ошибке — `data.json` **не затирается**, скрипт завершается с ненулевым кодом и понятным сообщением.
5. **Лог-сводка** в конце: счётчики проектов/задач/кураторов, что сматчилось/не нашлось (частично закрывает
   «быструю» автоматизацию — лог прогона).

### R4 — Безопасный deploy

В `deploy.sh`:

- `git add .` (строка 46) → **явный список**: `git add data.json` (+ изменённые отслеживаемые исходники,
  если переданы/нужны — `index.html`, `CLAUDE.md`, `docs/`, `*.py` по необходимости). `.txt`/`.xlsx` уже в `.gitignore`.
- **Guard перед коммитом:** `data.json` существует, непуст и валиден как JSON (`python3 -c json.load`); если
  не изменился (`git diff --quiet -- data.json`) — пропустить коммит/пуш с сообщением, без падения.
- **Guard пушей:** удалённые `upstream`/`origin` существуют (`git remote get-url`) — иначе понятное сообщение
  вместо сырого фейла `git`.
- `set -e` сохраняется; добавляются явные проверки с человекочитаемыми сообщениями.

### R5 — Экранирование строк в HTML

В `index.html` добавляем хелпер `escapeHTML(s)` (`&`, `<`, `>`, `"`, `'`) и применяем ко всем вставкам
строк из `data.json` в template-literals: `p.name`, `t.theme`, `executor`/`owner`-имена, `goal`, `indicators`,
`status`, строки `current_status` (1640-1648), `data-name` (1106, 1147 — заменить `replace(/"/g,...)` на `escapeHTML`).
URL остаётся как есть (число `id`), `href` собирается из `config.redmine_base`.

### Q1 — Smoke-тесты пайплайна

Каталог `tests/` + `pytest`:

- **Юнит-тесты чистых хелперов** (импорт без запуска `main` — гард `__main__` уже есть):
  `get_urgency`, `short_name`, `curator_key`, `canon_name`, `clean_pct`, `clean_date`, `normalize_col`/`resolve_column`
  (из `extract_data.py`/`config.py`); `parse_deadline`, `plural_days/tasks/projects` (из `overdue_report.py`).
- **Интеграционный smoke:** строим маленький Excel-фикстур (`openpyxl`) с 1-2 проектами, задачами, кураторами →
  прогон `extract()` во временной папке → проверки формы `data.json`: ключи верхнего уровня
  (`updated_at, summary, projects, all_tasks, curators, config`), наличие обязательных полей у объектов,
  корректность счётчиков `summary`.
- **Тест валидации R1:** битый фикстур (нет колонки / пусто) → `extract()` падает с ненулевым кодом и
  **не перезаписывает** существующий `data.json`.

### Q2 — Линтер + type hints

- `pyproject.toml` с конфигом `ruff` (разумный набор правил, длина строки, исключения).
- Type hints — на хелперы и сигнатуры публичных функций (`-> dict`, `-> str | None` и т.п.); без фанатизма,
  цель — ранняя ловля ошибок, не 100%-покрытие.

## 5. Обработка ошибок (общий подход)

- Python-скрипты: «падать рано и понятно» — структурные/sanity-ошибки бросают исключение с человекочитаемым
  текстом (что не так, что ожидалось, что нашли) и завершаются ненулевым кодом. Старые данные не затираются.
- `deploy.sh`: каждый внешний шаг (extract/report/overdue/git) проверяется; при сбое — понятное сообщение,
  пуш не выполняется на кривых данных.
- `index.html`: `escapeHTML` исключает поломку разметки; существующий `try/catch` в `loadData` сохраняется.

## 6. Тестирование и приёмка

- `pytest` зелёный локально; `ruff check` без ошибок.
- Контрольный прогон `deploy.sh` на текущих файлах даёт **тот же `data.json`** (поведение не изменилось),
  плюс новый ключ `config`. Дашборд открывается и показывает те же цифры (36/613/5, 17 просрочено, 122%).
- Тест валидации: битый фикстур не затирает `data.json`.

## 7. Риски и обратная совместимость

- **Риск:** новый ключ `config` в `data.json` — старые версии `index.html` его игнорируют (безопасно);
  новый `index.html` с фолбэком работает и со старым `data.json` без `config`.
- **Риск:** перевод колонок на резолвер может изменить, какая именно колонка выбирается. Митигируется
  тестом «тот же `data.json`» на реальных файлах перед коммитом.
- **Риск:** type hints/ruff могут вскрыть существующие огрехи — правим точечно, без рефакторинга поведения.
