# Transformation Dashboard

Дашборд мониторинга проектов трансформации. Данные извлекаются из Excel-файлов в `data.json`, который отображается в `index.html`.

## Роли в работе над проектом

- **Claude.ai (этот проект)** — планирование, концепция, постановка задач, обсуждение логики
- **Claude Code** — написание и изменение кода, обновление `CLAUDE.md` после каждой сессии

### Регламент обновления CLAUDE.md

После каждой рабочей сессии Claude Code **обязан**:
1. Дописать в `## История изменений` все изменения этой сессии (дата + краткое описание)
2. Обновить затронутые разделы документации (структура данных, формат сообщений, логика)
3. Закоммитить `CLAUDE.md` вместе с изменёнными файлами одним коммитом

Формат записи в истории: `| ГГГГ-ММ-ДД | Файл: описание изменения |`

## Архитектура

```
extract_data.py   →   data.json   →   index.html
(Python-скрипт)       (данные)        (дашборд, статический HTML)
```

## Файлы

| Файл | Назначение |
|------|-----------|
| `extract_data.py` | Скрипт извлечения данных из Excel → JSON |
| `data.json` | Готовые данные для дашборда (коммитится в git) |
| `index.html` | Дашборд (читает data.json через fetch) |
| `Dockerfile` | Контейнер для деплоя |
| `amvera.yml` | Конфигурация деплоя на Amvera |
| `deploy.sh` | Скрипт одной командой: extract → report → git commit → push (аргументы: `$1` — xlsx, `$2` — md) |

## Источники данных (Excel, в git не хранятся)

| Файл | Содержимое |
|------|-----------|
| `issues.xlsx` / `issues (1).xlsx` | Выгрузка из Redmine (трекеры: Паспорт проекта, Мероприятие проекта) |
| `ШТАТКА_ДБ.xlsx` | Штатная численность по кураторам |
| `ПРОЕКТЫ_Данные по высвобождению.xlsx` | Данные по высвобождению (шт.ед., часы) |

## Обновление данных

**Быстрый вариант** (одна команда):
```bash
./deploy.sh issues_08.04.xlsx ОТЧЕТ_08_04.md ОТЧЕТ_02_04.md
# $1 — xlsx-файл Redmine, $2 — md-файл отчёта, $3 — предыдущий md для diff (необязателен)
```
Скрипт: `extract_data.py` → `process_report.py [--prev $3]` → `git commit data.json` → `git push upstream main` + `git push origin main`.

**Вручную:**
1. Положить актуальные `.xlsx` файлы в папку проекта (в git не попадут — в `.gitignore`)
2. Запустить скрипт:
   ```bash
   python3 extract_data.py
   # или с явным именем файла Redmine:
   python3 extract_data.py "issues (1).xlsx"
   ```
3. Закоммитить обновлённый `data.json`:
   ```bash
   git add data.json
   git commit -m "update data.json"
   git push
   ```

## Структура data.json

```
{
  updated_at  — дата/время генерации
  summary     — сводная статистика
  projects    — список проектов (из трекера "Паспорт проекта")
  all_tasks   — все задачи всех уровней (итеративный обход L1–Ln)
  curators    — данные по кураторам (штатка + высвобождение)
}
```

Поля проекта `projects[]` (дополнительные):
```
url            — ссылка на проект в Redmine (из xlsx высвобождения)
internal_hours — внутреннее высвобождение, часы (из xlsx)
external_hours — внешнее высвобождение, часы (из xlsx)
total_units    — высвобождение всего, шт.ед. (из xlsx)
closed_at      — дата закрытия (колонка «Закрыта» из Redmine)
defense_at     — дата защиты (колонка «Дата и время защиты» из Redmine)
```

## Логика сущностей

- **Проект** (`owner` = Ответственный, `manager` = Назначена)
- **Этап/Подзадача** (`executor` = Назначена)
- **Срочность задач**: `overdue` / `today` / `urgent` (≤5д) / `soon` (≤14д) / `ok`
- **Закрытые статусы**: `Закрыта`, `Закрыто`, `Выполнено`, `Выполнена`, `Завершена` — все приравниваются к закрытым; статус `Выполнено` исключается из счётчиков просрочки наравне с `Закрыта`
- **`computeTaskKPIs`**: использует `urgency` + `allTasks2026` (те же данные, что вкладка Задачи) для синхронизации KPI между вкладками; overdue подсчитывается через `t.urgency === 'overdue'`, а не через сравнение дат

## Кураторы и высвобождение

Порядок групп в файле НМА: `Кренёва АА → Родевальд СЕ → Гуляев ВА`
`Комитет и РЦТ` = сумма всех трёх групп
Формула %: `высвобождение шт.ед. / план_20% × 100`

## Блок «Высвобождение %» (вкладка Обзор)

### Layout
Блок находится в правой колонке `.two-col`, внутри `.section-card`, над «Дедлайны по неделям».
Внутренняя сетка — `#vysvGrid` с классом `.vysv-grid` (grid 1fr 1fr, gap 12px).

### Структура рендера
1. **Панель детализации** (`#vysvDetailPanel`, `grid-column:1/-1`) — всегда первой, закреплена сверху.
   Контент обновляется при клике, сам элемент не пересоздаётся.
2. **Карточки кураторов** (compact) — только gauge-полукруг + имя, без данных в теле карточки.
   Всегда все видны, не скрываются и не dim-ятся.
   Активная карточка получает border + box-shadow через класс `.highlighted`.

### Поведение
- По умолчанию: выбран «Комитет и РЦТ», его данные в панели сверху.
- Клик по карточке → highlight переходит на неё, панель обновляется.
- `ownerFilter != 'all'` → highlight и панель переключаются на соответствующего куратора (матчинг по фамилии).

### Ключевые функции JS
- `renderVysvBlock(selectedName)` — главная функция; рендерит структуру один раз, затем только обновляет highlight и панель.
- `_vysvDetailHTML(curator)` — возвращает innerHTML панели: блок «Кадры» (2×3) + блок «Высвобождение» (2 ячейки) + акцентная строка.
- `highlightGauge(ownerFilter)` — обёртка, матчит owner → curator и вызывает `renderVysvBlock`.

### CSS-классы
- `.vysv-grid` — `display:grid`, `1fr 1fr`, `gap 12px`, `padding 16px`
- `.vysv-span2` — `grid-column: 1/-1`
- `.vysv-detail` — панель детализации (фон, border accent, border-radius)
- `.vysv-detail-accent` — акцентная строка «% высвобождения»
- `.highlighted` — highlight активной карточки (border + glow)

## Блок «Высвобождение %» — панель детализации (обновлено 2026-04-03)

### Структура `_vysvDetailHTML(curator)`

**Блок «Кадры»** — сетка 2 колонки, 3 строки:
- Штат (`headcount`) | ККП Штат/Факт (`kkp` / `kkp_fact`) — две строки внутри ячейки
- Итого факт (`fact_total`) | РЦТ Штат/Факт (`rct` / `rct_fact`) — две строки внутри ячейки
- План −20% (`plan_minus20` шт.ед.) | Вакансии (`vacancies`)

**Блок «Высвобождение»** — сетка 2 колонки:
- Внутреннее: `vysv_internal_hours` ч + `(/ 1972).toFixed(1)` шт.ед. (стопкой, часы крупно)
- Внешнее: `vysv_external_hours` ч + `(/ 1972).toFixed(1)` шт.ед. (стопкой)
- `null` → «—»

**Акцентная строка** (`vysv-detail-accent`):
- Слева: `pct_vysv`% крупно; цвет: `=== 100` → `--warn`, `> 100` → `--accent3`, `< 100` → `--danger`
- Справа стопкой: «План `plan_minus20` шт.ед. / Факт `vysv_units` шт.ед.»

**Антипереполнение:** все ячейки имеют `min-width:0; overflow:hidden; word-break:break-word`.

### Новые поля кураторов в `data.json`
- `vysv_internal_hours` — колонка «Внутрнее высвобождение» из НМА-файла (NaN-строки)
- `vysv_external_hours` — колонка «Внешнее высвобождение, часы»
- Для «Комитет и РЦТ» — суммируется по трём группам (`CURATOR_ORDER`)

### Таблица «Проекты — Готовность» (вкладка Обзор)
Цветовая маркировка строк по кварталу дедлайна (`border-left` + `rgba` фон):
- `row-q1` Q1 (янв–март): `#7c6af7` (фиолетовый, `--accent2`)
- `row-q2` Q2 (апр–июнь): `#00e5ff` (циан, `--accent`)
- `row-q3` Q3 (июл–сент): `#00ff9d` (зелёный, `--accent3`)
- `row-q4` Q4 (окт–дек):  `#ffb800` (жёлтый, `--warn`)

Бейджи `Q1: N … Q4: N` в `#qBadgesRow` (section-header), обновляются в каждом вызове `renderProjTable()`.

## Еженедельный отчёт (`process_report.py`)

Скрипт обрабатывает еженедельный `.md`-файл с отчётами по проектам.

### Запуск

```bash
python3 process_report.py ОТЧЕТ_01_04.md
python3 process_report.py ОТЧЕТ_01_04.md --telegram-only   # только файлы Telegram
python3 process_report.py ОТЧЕТ_01_04.md --no-telegram     # только обновить data.json
python3 process_report.py ОТЧЕТ_01_04.md --data data.json  # явный путь к data.json
python3 process_report.py ОТЧЕТ_08_04.md --prev ОТЧЕТ_02_04.md  # diff с предыдущим
```

### Что делает

1. Парсит `.md`-файл (блоки `✅ Выполнено` и `📍 Текущие этапы`)
2. Сопоставляет проекты с `data.json` — по ID из URL `🔗`, резерв: нечёткое совпадение имён (`difflib`, порог 0.6)
3. Пишет `current_status` в найденные проекты
4. Делает резервную копию `data.json.bak`
5. Генерирует **два файла** Telegram-сообщений

### Выходные файлы

| Файл | Содержимое |
|------|-----------|
| `telegram_priority_ДД_ММ_ГГГГ.txt` | Приоритетные проекты (`is_priority=true`) |
| `telegram_transform_ДД_ММ_ГГГГ.txt` | Трансформационные проекты (`is_priority=false`) |

### Формат Telegram-сообщения

- Заголовок группы со счётчиком: `🔴 ПРИОРИТЕТНЫЕ ПРОЕКТЫ (3)`
- Нумерация проектов внутри группы: `1. 🔴 Электронные ДС с ПИК`
- Сортировка по возрастанию дедлайна
- Секция `⚠️ ИНФОРМАЦИЯ НЕ ПРЕДСТАВЛЕНА` (если есть активные проекты без отчёта)
- Без Markdown-символов (`*`, `_`) — текст готов к ручному копированию в Telegram

Формат одного проекта:
```
1. 🔵 Название проекта
👤 Кренёва А.А.
🔗 https://transformation.rm.mosreg.ru/#/issues/12345
📉 Высвобождение:
   Внутреннее: 9920 ч / 5.03 шт.ед.
   Внешнее:    43340 ч / 21.98 шт.ед.

✅ Выполнено:
...

📍 В работе:
...
─────────────────
```

Логика полей:
- `👤` — `owner_short` из `data.json` (Ответственный из Redmine, совпадает с фильтром дашборда); резерв — `person` из строки `👤` в MD-файле
- `🔗` — URL из строки `🔗` в MD-файле или поле `url` из `data.json`
- `📉 Высвобождение` — строка выводится только если часы > 0; шт.ед. = часы / 1972; блок скрывается целиком если все значения None или 0

### Признак приоритетности

Поле `is_priority` в `data.json` — извлекается из колонки **«Приоритетный проект»** файла Redmine.
`is_priority = true` если значение `'Да'`.
При ручном редактировании `data.json` поле можно проставить напрямую.

### Сопоставление проектов (URL → ID)

Поддерживаются форматы URL в строке `🔗`:
- `https://transformation.rm.mosreg.ru/#/issues/11498` (стандартный)
- `https://transformation-back.rm.mosreg.ru/issues/11245` (без `#`)
- Если URL содержит `/projects/` (не `/issues/`) — сопоставление по нечёткому имени (`difflib`, порог 0.6)

### Diff-режим (`--prev`)

Флаг `--prev PREV_REPORT` включает сравнение текущего отчёта с предыдущим.

**Что меняется в выводе:**
- Шапка сообщений: `🔄 Сравнение с отчётом от ДД.ММ.ГГГГ`
- Строки «В работе», которых не было в предыдущем «В работе» → префикс `🆕`
- Строки «Выполнено», которых не было ни в «Выполнено» ни в «В работе» предыдущего → `🆕`
- Проект, отсутствующий в предыдущем отчёте → суффикс `🆕 новый проект` в заголовке

**Вспомогательные функции diff:**

| Функция | Назначение |
|---------|-----------|
| `strip_dates(s)` | Удаляет `(дд.мм)` / `(дд.мм.гггг)` перед сравнением |
| `normalize_line(s)` | Убирает `•`, `-`, `1.`, лишние пробелы; lowercase |
| `_text_to_norm_set(text)` | Текст → множество нормализованных строк |
| `compute_new_lines(cur, prev_comp, prev_cur)` | Строки текущего блока, отсутствующие в предыдущем |
| `find_prev_project(proj, prev_index)` | Матчинг по `issue_id`, резерв — нечёткое имя |
| `build_prev_index(prev_groups)` | Строит `{by_id, by_name}` из разобранного предыдущего отчёта |
| `render_block_with_diff(block_text, new_lines_set)` | Вставляет `🆕 ` перед новыми строками |

Без `--prev` поведение идентично прежнему.

### Парсер `📍`-строк

Поддерживаемые форматы заголовка блока «В работе» в MD-файле:
- `📍 Текущие этапы (в работе):`
- `📍 Текущие этапы:`
- `📍 В работе:`

Все три стрипаются — в `current` попадает только тело блока без заголовка.

### Дашборд и current_status

`index.html` читает `d.current_status` для блока «Статус проекта» (вкладка Детализация).
После запуска `process_report.py` данные сразу отображаются на дашборде.
