# Transformation Dashboard

Дашборд мониторинга проектов трансформации. Данные извлекаются из Excel-файлов в `data.json`, который отображается в `index.html`.

## Роли в работе над проектом

- **Claude.ai (этот проект)** — планирование, концепция, постановка задач, обсуждение логики
- **Claude Code** — написание и изменение кода, обновление `CLAUDE.md` после каждой сессии

### Регламент обновления CLAUDE.md

После каждой рабочей сессии Claude Code **обязан**:
1. Дописать в `## История изменений` все изменения этой сессии (дата + краткое описание)
2. Обновить затронутые разделы документации (структура данных, формат сообщений, логика)
3. Закоммитить `CLAUDE.md` вместе с изменёнными файлами одним коммитом

Формат записи в истории: `| ГГГГ-ММ-ДД | Файл: описание изменения |`

## Архитектура

```
extract_data.py   →   data.json   →   index.html
(Python-скрипт)       (данные)        (дашборд, статический HTML)
```

## Источники данных (Excel, в git не хранятся)

| Файл | Содержимое |
|------|-----------|
| `issues.xlsx` / `issues (1).xlsx` | Выгрузка из Redmine (трекеры: Паспорт проекта, Мероприятие проекта) |
| `ШТАТКА_ДБ.xlsx` | Штатная численность по кураторам |
| `ПРОЕКТЫ_Данные по высвобождению.xlsx` | Данные по высвобождению (шт.ед., часы) |

## Обновление данных

1. Положить актуальные `.xlsx` файлы в папку проекта (в git не попадут — в `.gitignore`)
2. Запустить скрипт:
   ```bash
   python3 extract_data.py
   # или с явным именем файла Redmine:
   python3 extract_data.py "issues (1).xlsx"
   ```
3. Закоммитить обновлённый `data.json`:
   ```bash
   git add data.json
   git commit -m "update data.json"
   git push
   ```

## Структура data.json

```
{
  updated_at  — дата/время генерации
  summary     — сводная статистика
  projects    — список проектов (из трекера "Паспорт проекта")
  stages      — этапы (дочерние задачи проектов)
  tasks       — подзадачи (дочерние задачи этапов)
  all_tasks   — stages + tasks вместе
  curators    — данные по кураторам (штатка + высвобождение)
}
```

Поля проекта `projects[]` (дополнительные, из xlsx высвобождения):
```
url            — ссылка на проект в Redmine (из колонки «Ссылка на акцептованную идею»)
internal_hours — внутреннее высвобождение, часы
external_hours — внешнее высвобождение, часы
total_units    — высвобождение всего, шт.ед.
```

## Логика сущностей

- **Проект** (`owner` = Ответственный, `manager` = Назначена)
- **Этап/Подзадача** (`executor` = Назначена)
- **Срочность задач**: `overdue` / `today` / `urgent` (≤5д) / `soon` (≤14д) / `ok`
- **Закрытые статусы**: `Закрыта`, `Закрыто`, `Выполнено`, `Выполнена`, `Завершена` — все приравниваются к закрытым; статус `Выполнено` исключается из счётчиков просрочки наравне с `Закрыта`
- **`computeTaskKPIs`**: использует `urgency` + `allTasks2026` (те же данные, что вкладка Задачи) для синхронизации KPI между вкладками; overdue подсчитывается через `t.urgency === 'overdue'`, а не через сравнение дат

## Кураторы и высвобождение

Порядок групп в файле НМА: `Кренёва АА → Родевальд СЕ → Гуляев ВА`
`Комитет и РЦТ` = сумма всех трёх групп
Формула %: `высвобождение шт.ед. / план_20% × 100`

## Еженедельный отчёт (`process_report.py`)

Скрипт обрабатывает еженедельный `.md`-файл с отчётами по проектам.

### Запуск

```bash
python3 process_report.py ОТЧЕТ_01_04.md
python3 process_report.py ОТЧЕТ_01_04.md --telegram-only   # только файлы Telegram
python3 process_report.py ОТЧЕТ_01_04.md --no-telegram     # только обновить data.json
python3 process_report.py ОТЧЕТ_01_04.md --data data.json  # явный путь к data.json
```

### Что делает

1. Парсит `.md`-файл (блоки `✅ Выполнено` и `📍 Текущие этапы`)
2. Сопоставляет проекты с `data.json` — по ID из URL `🔗`, резерв: нечёткое совпадение имён (`difflib`, порог 0.6)
3. Пишет `current_status` в найденные проекты
4. Делает резервную копию `data.json.bak`
5. Генерирует **два файла** Telegram-сообщений

### Выходные файлы

| Файл | Содержимое |
|------|-----------|
| `telegram_priority_ДД_ММ_ГГГГ.txt` | Приоритетные проекты (`is_priority=true`) |
| `telegram_transform_ДД_ММ_ГГГГ.txt` | Трансформационные проекты (`is_priority=false`) |

### Формат Telegram-сообщения

- Заголовок группы со счётчиком: `🔴 ПРИОРИТЕТНЫЕ ПРОЕКТЫ (3)`
- Нумерация проектов внутри группы: `1. 🔴 Электронные ДС с ПИК`
- Сортировка по возрастанию дедлайна
- Секция `⚠️ ИНФОРМАЦИЯ НЕ ПРЕДСТАВЛЕНА` (если есть активные проекты без отчёта)
- Без Markdown-символов (`*`, `_`) — текст готов к ручному копированию в Telegram

Формат одного проекта:
```
1. 🔵 Название проекта
👤 Кренёва А.А.
🔗 https://transformation.rm.mosreg.ru/#/issues/12345
📉 Высвобождение:
   Внутреннее: 9920 ч / 5.03 шт.ед.
   Внешнее:    43340 ч / 21.98 шт.ед.

✅ Выполнено:
...

📍 В работе:
...
─────────────────
```

Логика полей:
- `👤` — `owner_short` из `data.json` (Ответственный из Redmine, совпадает с фильтром дашборда); резерв — `person` из строки `👤` в MD-файле
- `🔗` — URL из строки `🔗` в MD-файле или поле `url` из `data.json`
- `📉 Высвобождение` — строка выводится только если часы > 0; шт.ед. = часы / 1972; блок скрывается целиком если все значения None или 0

### Признак приоритетности

Поле `is_priority` в `data.json` — извлекается из колонки **«Приоритетный проект»** файла Redmine.
`is_priority = true` если значение `'Да'`.
При ручном редактировании `data.json` поле можно проставить напрямую.

### Сопоставление проектов (URL → ID)

Поддерживаются форматы URL в строке `🔗`:
- `https://transformation.rm.mosreg.ru/#/issues/11498` (стандартный)
- `https://transformation-back.rm.mosreg.ru/issues/11245` (без `#`)
- Если URL содержит `/projects/` (не `/issues/`) — сопоставление по нечёткому имени (`difflib`, порог 0.6)

### Дашборд и current_status

`index.html` читает `d.current_status` для блока «Статус проекта» (вкладка Детализация).
После запуска `process_report.py` данные сразу отображаются на дашборде.
